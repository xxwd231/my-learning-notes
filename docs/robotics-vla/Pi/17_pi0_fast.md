## 一、 FAST 的三层核心贡献

-   **算法层**
    
    -   **DCT (离散余弦变换)** 把平滑动作曲线压缩约 10x。
        
    -   **BPE (字节对编码)** 进一步学习动作“惯用语”。
        
    -   将 **350 个浮点数** 成功压缩成 **30–60 个 Token**，且完整保留了动作的时序结构。
        
-   **工程层**
    
    -   把机器人控制完全统一到标准 LLM 训练管道中。
        
    -   训练速度大幅提升 **5x**，且大语言模型（LLM）现有的所有优化技巧（如 FlashAttention、KV Cache 等）均可直接复用。
        
-   **认知层**
    
    -   证明了“仅对 Tokenization 进行小改动”就能带来 **5x** 的效率提升。
        
    -   说明 VLA (Vision-Language-Action) 模型的训练远远未被充分优化，各个环节均隐藏着巨大的提升空间。
        

## 二、 FAST 完整处理流程 (5 步详解)

**初始输入**：7 关节 $\times$ 50 帧 = 350 个浮点数（表示一段典型机械臂手部运动轨迹，7 个 DoF 在未来 50 帧内的预测位置）。如果直接让大模型预测 350 个浮点数，会导致序列过长、计算极其缓慢，无法满足极低延迟要求。

### 步骤 1：归一化 (Normalization)

-   **操作**：将各个关节的角度值独立缩放到 $[-1, 1]$ 目标区间。
    
-   **目的**：消除不同关节物理量纲的巨幅差异（如底座关节可达 180°，而手腕可能只有 30°），让模型在统一的数值尺度下学习，使训练更稳定。
    

### 步骤 2：对每个关节做 DCT (离散余弦变换)

-   **操作**：将时间维度的 50 帧轨迹，转换为频率维度的 50 个频率分量（形成 $7 \times 50$ 的频率矩阵）。
    
-   **目的**：**FAST 最核心的思想**。机器人的运动通常是平滑连续的，能量主要集中在低频部分，高频往往只是细小抖动或噪音。DCT 成功将轨迹特征从“时间域”转到了“频域”。
    

### 步骤 3：量化 (Quantization: 浮点 $\rightarrow$ 整数)

-   **操作**：将 DCT 产生的浮点数乘以一个系数（如图片中的 100）并四舍五入变成整数（例：$3.20 \rightarrow 320$, $-1.48 \rightarrow -148$）。
    
-   **目的**：大语言模型的词表（Vocabulary）全都是离散的整数 ID。这一步将连续的物理信号彻底转化为离散的“词汇”，为后续的 NLP 化处理打下基础。
    

### 步骤 4：按“低频优先”顺序展开，截断高频

-   **操作**：按频率从低到高将矩阵拉平为一维数组，并**直接砍掉高频部分**。
    
-   **结果**：从原本的 350 个数字，暴降截断压缩到了约 **40 个整数**。
    
-   **目的**：低频包含了动作的“主要轮廓”，扔掉高频（抖动/噪音）几乎不影响机器人的实际执行精度，却将模型的预测负担降低了近 90%，大幅降低了自回归预测（Next-token prediction）的推理延迟。
    

### 步骤 5：BPE 编码 (Byte Pair Encoding)

-   **操作**：使用 NLP 中最经典的 BPE 算法，寻找经常一起出现的“整数对”，将其合并为一个全新的、更高级的 Token。
    
-   **目的**：进一步压缩序列长度，最终将 50 帧动作压缩为仅 **30–60 个 Token**，让这些动作组合变成了类似于“词语”的典型动作范式。
    

## 三、 深入理解 BPE 动作词表构建

BPE（与 GPT 分词器原理完全相同）的核心思想是：**反复找出序列中最高频的相邻数字对，将其合并为一个新符号**。

-   **第一步 (初始状态)**：每个整数独立作为原始 Token。
    
    -   `[..., 320, 185, 94, -148, -87, 320, 185, 94, -148, ...]`
        
-   **第二步 (统计高频对并合并)**：
    
    -   统计发现 `"320, 185"` 出现频率最高（如 8 万次），新增符号 `#A` 替换所有出现位置：
        
    -   `[..., #A, 94, -148, -87, #A, 94, -148, ...]`
        
-   **第三步 (迭代合并)**：
    
    -   继续寻找最高频对 `" #A, 94 "`（如 5 万次），新增符号 `#B` 替换：
        
    -   `[..., #B, -148, -87, #B, -148, ...]`
        
-   **总结**：重复数万次后，词表里的每个 Token 都对应一段常见动作的频率模式，将序列从 ~40 个整数进一步压缩到 **30–60 个 Token**。
    

## 四、 局限性分析：推理速度瓶颈

FAST 虽然大幅减少了 Token 数量，但由于采用了自回归 Token 预测，引入了串行生成的瓶颈：

-   **Flow Matching ($\pi_0$)**：
    
    -   **并行生成**：从输入噪声经 10 步迭代，一次性并行输出全部 350 维动作（耗时约 27ms）。
        
-   **FAST ($\pi_0$-FAST)**：
    
    -   **逐 Token 串行生成**：第 $i$ 个 Token 的预测依赖前 $i-1$ 个 Token（自回归本质）。
        
    -   输出 30–60 个 Token 需要逐个串行解码，耗时约 **30–50ms**。
        
-   **工程解决思路**：团队认为自回归加速在 LLM 领域已有极其成熟的解决方案（如 **KV Cache**、**Speculative Decoding (投机采样)**），这是一个完全已知且可在工程上解决的问题。
    

## 五、 控制方案对比（列表版）

### 1. 简单分词 (Direct Quant)

-   **序列长度**：350 tokens
    
-   **时序连续性**：无
    
-   **精细任务**：失败
    
-   **训练速度**：慢
    
-   **推理延迟**：慢
    
-   **LLM 生态兼容**：部分
    

### 2. Flow Matching ($\pi_0$)

-   **序列长度**：不适用（连续空间）
    
-   **时序连续性**：有
    
-   **精细任务**：成功
    
-   **训练速度**：慢
    
-   **推理延迟**：**快 (~27ms)**
    
-   **LLM 生态兼容**：否（需额外 Action Expert）
    

### 3. FAST ($\pi_0$-FAST)

-   **序列长度**：**30–60 tokens**
    
-   **时序连续性**：**有 (DCT 保证)**
    
-   **精细任务**：**成功**
    
-   **训练速度**：**快 5x**
    
-   **推理延迟**：较慢 (~30-50ms)
    
-   **LLM 生态兼容**：**完全兼容**
    

_(注：后续研究如 Knowledge Insulation 尝试将 FAST 的高兼容性与 Flow Matching 的并行高速推理优势结合，兼顾两者长处)。_
